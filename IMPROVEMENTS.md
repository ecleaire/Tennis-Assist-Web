# 改善候補

## UI / UX
- 記録画面の中間結果表は列数が多いため、スマホではカード型表示に切り替えると確認しやすくなる。
- 「保存」「同意」「確定」などの重要操作は、現在の色分けに加えてアイコンや短い補助文を併用すると誤操作をさらに減らせる。
- 試合終了画面は読み上げ用と記録確認用で情報密度を切り替えられると、審判と選手の両方に使いやすい。

## スマホ縦画面
- 記録入力のボール数選択を大きめのステッパー形式にすると、屋外や慌ただしい進行中でも押し間違いが減る。
- ルール画面と履歴画面は、縦画面ではフィルターを折りたたみ式にすると表示領域を広く使える。

## PWA / オフライン
- Service Worker は現在アプリ本体を一括キャッシュしている。将来的には初回起動に必須のファイルと後追いキャッシュを分けると、初回表示が軽くなる。
- 更新通知をアプリ内に出し、ユーザーが明示的に再読み込みできる導線を用意すると、古いキャッシュによる混乱を減らせる。

## CSV
- CSVインポート時に列不足や形式違いを検出し、読み込めた件数とスキップした理由を表示すると運用時の安心感が上がる。
- 大会進行用に「確定済み試合結果だけ」を出力する軽量CSVを追加すると、表計算で扱いやすい。

## パフォーマンス
- Godot Web Export の `index.wasm` が大きいため、初回ロードの主な重さはエンジン本体にある。さらに軽くするなら、ニュース・リンク・ルールなど静的画面をHTML側に寄せる案がある。
- 起動直後に全画面分のシーンを生成している場合は、初回表示に必要な画面だけ生成し、他画面は初回遷移時に生成する設計へ変えるとメモリと描画負荷を下げられる。
- フォントは現在 Web Export 除外設定で多くを外しているが、実際に使うウェイトをさらに絞る余地がある。

## データ保存
- 端末内保存はプライバシー面で扱いやすい一方、端末紛失やブラウザデータ削除に弱い。CSVバックアップ導線をより目立たせると安全。
- 履歴削除やCSVインポートの直後に自動バックアップを提案するUIを追加できる。

## 保守性
- 記録画面の責務が大きくなっているため、履歴表示、チーム編集、最終確認、CSV入出力を小さなスクリプトへ分けると変更しやすい。
- ボタン配色の定義は各画面に重複している。共通テーマ関数へ寄せると色変更が一箇所で済む。

## 将来的な拡張性
- サーバー共有を追加しない前提でも、Google Apps Script へ「確定済み試合結果だけを送信」する任意機能は拡張候補になる。
- その場合は、オフライン時の送信待ち、重複送信防止、送信先URLの保存方法、誤送信の取り消し方法を先に設計する必要がある。
